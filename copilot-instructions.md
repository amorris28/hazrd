## Quick orientation

This repository is an R package named `hazrd` focused on evaluating and
visualizing polygenic hazard scores (PHS) using survival models. Key
components live in the `R/` directory and tests use `testthat/`.

Files to scan first - `R/phs_metrics.R` — user-facing metric
orchestrator (HR, C-index, OR) - `R/metrics-internal.R` — metric
implementations (.\_calc\_\* functions) - `R/bootstrap-internal.R` —
bootstrapping orchestration; uses
[`boot::boot`](https://rdrr.io/pkg/boot/man/boot.html) -
`R/create_lookup.R` — generation of per-percentile K-M predictions -
`tests/testthat/` — unit tests to validate behavior - `DESCRIPTION`,
`NAMESPACE`, `README.Rmd` — packaging and install metadata - **Never
edit `README.md` directly** — it is auto-generated by knitting
`README.Rmd` via
[`rmarkdown::render()`](https://pkgs.rstudio.com/rmarkdown/reference/render.html).
Always use the Docker container which has pandoc:
`docker run --rm -v /path/to/hazrd:/repo my-r-image \ Rscript -e 'rmarkdown::render("/repo/README.Rmd", output_file = "/repo/README.md")'`
Do NOT use [`knitr::knit()`](https://rdrr.io/pkg/knitr/man/knit.html) —
it lacks pandoc and emits the YAML frontmatter as literal text in the
output. - `docs/specs/spec-sheet.md` — package spec sheet (may be
local/ignored; ask the maintainer for a copy)

Big-picture architecture & data flow - Input: a data.frame with at
minimum `phs`, `age` (time) and `status` (event). -
[`phs_metrics()`](https://amorris28.github.io/hazrd/reference/phs_metrics.md)
is the single primary entry-point for computing metrics; it fits a Cox
model ([`survival::coxph`](https://rdrr.io/pkg/survival/man/coxph.html))
and derives metrics from the fit. - Metric calculation is split: point
estimates are computed first; if `bootstrap = TRUE` the package calls
`.run_bootstrap()` which re-fits the model on resamples and derives
bootstrap CIs using
[`boot::boot`](https://rdrr.io/pkg/boot/man/boot.html) + `boot.ci`. -
Outputs are tidy tibbles with canonical columns: `metric`, `estimate`,
`conf_low`, `conf_high`, `se`, `n_numerator`, `n_denominator`, `method`,
`adjusted`.

Developer spec - See `docs/specs/spec-sheet.md` for the canonical
package specification. NOTE: this file may be kept locally and
git-ignored by the maintainer. If you cannot find it in the repo, ask
the maintainer to provide the Markdown copy. The spec documents package
goals, input conventions (string column names), metric naming
conventions (e.g. `HR[80-100]_[0-20]`), bootstrap behavior (internal
[`boot::boot`](https://rdrr.io/pkg/boot/man/boot.html) usage and CI
methods), and worked examples for `phs_metrics`, `phs_km`, and
`phs_abs_risk`.

Project-specific conventions & patterns (important for edits) - Column
names are usually passed as strings (e.g. `phs = "phs"`, `time = "age"`,
`event = "status"`). Functions accept string names and often index the
data with `data[[phs]]` rather than non-standard-eval. - HR metric
naming convention is literal and parsed elsewhere: e.g.
`HR[80-100]_[0-20]` — tests and bootstrapping rely on these exact names
to align CI results back to rows. - Bootstrapping: `parallel` argument
is present but only `"no"` is implemented (see `.run_bootstrap()`
guard). Avoid changing parallel semantics without updating that guard
and tests. - Many internal helpers return tibbles with the canonical
column order; keep that contract when adding new metrics.

Build / test / debug workflows (commands you can run locally) - Install
dev version: in R: `devtools::install("/path/to/hazrd")` or
`devtools::build()` then
`install.packages(..., repos = NULL, type = "source")`. - Run package
tests: in R: `devtools::test()` or from shell: R -e ‘devtools::test()’ -
Run R CMD check: from repo root: R CMD check –as-cran . - The repo has a
GitHub Action `R-CMD-check.yaml` (see badge in `README.md`) — mimic the
action’s R versions if reproducing CI locally.

Integration points & external deps - Declared imports in `DESCRIPTION`:
`boot`, `dplyr`, `ggplot2`, `survival`, `tibble`, `tidyr`, `scales`. Use
those for consistent idioms. -
[`boot::boot`](https://rdrr.io/pkg/boot/man/boot.html) and
[`boot::boot.ci`](https://rdrr.io/pkg/boot/man/boot.ci.html) are central
— changes to the bootstrap interface must preserve the numeric-vector
contract used by `boot`.

Testing & safety notes for code changes - Add unit tests to
`tests/testthat/` for any change in metric logic or bootstrap behavior.
The package uses `testthat` edition 3. - Keep the reported metric names
stable: tests and `bootstrap` matching rely on exact metric name
strings. - When modifying exported functions, update roxygen comments in
`R/` source — run `devtools::document()` to regenerate `NAMESPACE` and
`man/` files. - **Never use `\%` in roxygen `#'` comments** — the
backslash doubles in the generated Rd (`\\%`), where `%` then starts an
Rd comment and can corrupt the `\arguments{}` block. Use plain text
(e.g. “percent” or “0-20”) instead.

## Pre-merge checklist (dev -\> main)

This is the full workflow from issue to merged release. Work is done on
`dev`; `main` should always be clean and passing.

### 1. During development (on `dev`)

Issues for the milestone are tagged in GitHub (e.g. milestone `0.2.2`)

All feature/fix commits are on `dev`

A `testthat` test covers every changed behavior

### 2. Documentation (on `dev`, before merge)

**roxygen**: update `@param`, `@return`, `@examples` in `R/` source for
any changed or new exported functions

- Never use `\%` in roxygen comments — use plain text (“percent” or
  “0-20”) instead; see safety notes above

**`devtools::document()`** — regenerate `NAMESPACE` and all `man/` Rd
files; commit any changed files in `man/` and `NAMESPACE`

**`NEWS.md`** — add an entry under the current version heading
describing new features, fixes, and breaking changes

**Vignette** (`vignettes/hazrd.Rmd`) — update examples and narrative to
reflect any new or changed parameters/behavior

**`README.Rmd`** — update quick example and installation section if
needed; then render via Docker:

    docker run --rm -v /path/to/hazrd:/repo my-r-image \
      Rscript -e 'rmarkdown::render("/repo/README.Rmd", output_file = "/repo/README.md")'

Commit the updated `README.md`

**`DESCRIPTION`** — bump `Version:` field to the new release number

### 3. Final checks (on `dev`, before merge)

**`devtools::check()`** — must pass with 0 errors, 0 warnings, 0 notes

**`devtools::test()`** — all tests green

Commit everything; confirm `git status` is clean

### 4. Merge and tag

`git checkout main && git merge --ff-only dev`

`git push origin main`

`git tag vX.Y.Z main && git push origin vX.Y.Z`

Create GitHub release pointing at the tag (use `gh release create`)

Close resolved issues (commit messages with `Closes #N` do this
automatically)

### 5. Post-merge

`git checkout dev` — continue future work on `dev`

Bump `DESCRIPTION` version to next dev version (e.g. `0.2.2.9000`) on
`dev`

Good example edits to get started - Small bugfix: ensure `or_age` bounds
are handled (see `.calc_or_metric`) — add test covering an `or_age`
outside observed range. - Add a new C-index method: implement `"uno"` in
`.calc_cindex_metric()` and add a test and example in `R/phs_metrics.R`
documentation.

Prompting hints for the AI agent - When you change metric output shape,
list the exact column names and show a one-row example tibble so
maintainers can verify compatibility. - For bootstrap changes, include a
minimal reproducible call of `phs_metrics` with `bootstrap = TRUE` and
`n_boot = 50` in tests so CI runs fast.
